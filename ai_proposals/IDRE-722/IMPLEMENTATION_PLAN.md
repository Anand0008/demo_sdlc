## Implementation Plan

**Step 1: Implement list generation logic for stuck disputes**  
Add a new exported function `generatePaymentStatusCorrectionLists` that queries `prisma.case.findMany`. It will evaluate the `paymentAllocations` for both the initiating and non-initiating parties to determine if one or both have fully paid their required amounts. It will filter for cases where one party has paid but the status is not `PENDING_SECOND_PAYMENT`, and cases where both have paid but the status is not `FINAL_ELIGIBILITY`. The function will return two arrays (List A and List B) containing the dispute number, current status, payment status per party, and the date of the last payment received.
Files: `lib/actions/case.ts`

**Step 2: Implement fix execution logic for confirmed disputes**  
Add a new exported function `executePaymentStatusCorrection` that accepts two arrays of confirmed case IDs (one for List A, one for List B). For each case ID in List A, update the case status to `PENDING_SECOND_PAYMENT`. For each case ID in List B, update the case status to `FINAL_ELIGIBILITY`. For every updated case, call the existing `logCaseAction` function (or create a `CaseAction` record) to log a note stating: 'Status corrected due to the two-payment feature rollout.' Ensure the updates are wrapped in a transaction or processed sequentially to avoid partial failures.
Files: `lib/actions/case.ts`
